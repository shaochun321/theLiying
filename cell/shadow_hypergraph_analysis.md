## 构想一：影子超图 — 逐条分析

### 可取之处 ✅

**1. "死亡不是终结，而是维度压缩"（卷缩机制）**

> 对这个子图进行一次快速的图谱分解，只提取它的主特征向量

这个想法**直接对应**当前代码库中已有的两个机制：

```python
# 已有: node_necropolis — 但只是把死节点扔进 SQLite
# 已有: Engine B 的 cumulative_potential — 但死亡后被丢弃

# 提案: 不要丢弃！压缩为特征向量后保留
```

**可取**: 当前的 `node_necropolis` 确实只是"打成字符串写入坟墓"，没有保留任何结构信息。如果改为保存死亡节点的 z_t 特征向量（7维），代价极低（7个浮点数），但保留了该结构在测度空间中的"位置"。

**2. "幽灵共振 = 余弦相似度 + 微弱偏置"**

> 将当前的 z_t 与影子层中的宏节点计算余弦相似度，注入微弱偏置

这与你刚才说的"保留一个稳固的赫布超图层，其他作为候选和路径"**完全一致**。具体来说：

- 稳固层 = `HebbianSignalTransform.W_signal`（冻结后不变）
- 影子层 = 历史 z_t 特征向量的集合
- 共振 = 新 z_t 与历史 z_t 的余弦相似度
- 偏置 = **不修改 W_signal**，只在审计日志中记录"与历史轨迹 X 相似度 = 0.87"

**关键**: 2026.5.15.2 正确指出了**单向性**：

> 影子层是绝对的单行道。影子永远不能直接修改主线的核心状态。

这完全符合你之前的判断 — 不让影子超图直接反馈到主线。

**3. 慢波睡眠周期（异步更新）**

> 影子图的更新必须放在独立的后台线程。每 1000 个 Tick 吸收一次残渣

这解决了你担心的"无限迭代"问题 — 影子层的时间尺度与主线完全解耦。

### 需要谨慎的 ⚠️

**"先验引力注入"**

> 向上层的主图注入一股极其微弱的"偏置电压"

2026.5.15.1 在这里太激进了。它说"注入偏置"，这就是我们刚才讨论过的反馈闭环风险。2026.5.15.2 修正了这一点，改为"只提供微弱的引力偏置，加速类似结构的再次形成"，但**即使是微弱的偏置也是反馈**。

**我的建议**: 在当前阶段，幽灵共振的输出**只写入审计表**（`shadow_resonance_log`），不注入任何偏置。工程师可以查看这些日志来决定是否手动调整 W_signal。

## 构想二：复数域权重 — 逐条分析

### 数学上优美，工程上过早 ⏳

2026.5.15.2 的裁决是正确的：**远期研究**。

**为什么过早：**

1. **当前系统还没有闭合赫布环**。在三套赫布系统都互不相通的现状下，给权重加复数维度只会增加混乱。
2. **可解释性成本太高**。你的项目一直强调"审计可追踪" — 116 张 SQLite 表的 provenance_hash 就是证据。复数相位旋转会严重破坏这个审计特性。
3. **实数域的双层架构可以模拟相同效果**。正如 2026.5.15.2 所说：

> 主超图的高频更新就是"实部"，影子超图的缓慢沉积就是"虚部"

所以你不需要 `W = a + bi`，你只需要 `W_active` + `W_shadow` 两张表。

### 唯一可取的洞察

> 在实数域里，0 就是彻底的死亡，遗忘就是物理意义上的抹除

这是真的。当前的 `w_floor = 0.01`（ABConfig）意味着权重低于 0.01 就等于死了。构想一的影子超图正好解决了这个问题 — 低于 0.01 的权重不被删除，而是被压缩后移入影子层。

## 构想三：对抗性生存装甲 — 逐条分析

### 1. 拓扑迟滞（Topological Hysteresis）✅ 最可取

> 不要使用瞬时的 Xin 来触发相变。引入"时间积分窗口（Leaky Integrate-and-Fire）"

**对照现有代码**:

```python
# Engine B 已经有 M_eff（有效拓扑质量）
# 但 M_eff 影响的是权重更新幅度，不是相变触发
# 缺的是: "积分→阈值→触发"机制
```

这直接可以映射到 `InternalMeasureTime`：

```python
# d_σ_t 本身就是一个"积分器"
# 可以用 d_σ_t 的滑动窗口积分来判定 Xin 是否足够"重"
# d_σ_t 积分 > 阈值 → 允许相变
# d_σ_t 积分 < 阈值 → 抑制相变（迟滞）
```

这与你的项目底色**完全一致** — 用 d_σ_t 物理测度而非数学阈值来控制相变。

**当前代码中已有的雏形**:

```python
# Engine B 的 M_eff 计算 (engine_b_topological_inertia.py):
# M(Φ) = M_min + (M_max - M_min) × sigmoid(α × Φ)
# 高 Φ → 高 M → 低 Δw → 结构更稳固

# 这已经是"拓扑迟滞"的雏形！
# 但它只影响权重更新速度，不影响相变触发
```

### 2. L1.5 休眠舱 ✅ 部分可取

> 将 P-Core 的 GMM 参数冻结，塞进 L1.5 缓存。用 Wasserstein 距离检测是否需要唤醒

**对照现有代码**:

```python
# XiDecayEngine 已经有 "held → decaying → proto_candidate → quarantined" 状态机
# 但这是 Xin 残差的生命周期，不是 P-Core 的

# GMM 的 mu/sigma 参数确实可以被冻结和保存
# OT 的 Wasserstein 距离已经可以用 — sliced_wasserstein() 10× faster
```

**可取**: P-Core 死亡前冻结其 z_t 特征向量（不是全部 GMM 参数 — 太重了），进入休眠。唤醒用余弦相似度（比 Wasserstein 快得多，且 z_t 只有 7 维）。

**需要谨慎**: "直接解冻拉回 L1" 太激进。应该是"发出唤醒候选信号，由工程师/高层决策"。

### 3. 元核顶替（Meta-Core）⚠️ 概念可取，实现复杂

> 自动生成一个父级节点，把频繁切换的子 P-Core 降级为自己的内部状态

这本质上是**层次聚类**。在 BayesianMotionRecognizer 中，当 regime 频繁切换（stationary → oscillation → stationary → ...），应该识别出"这本身就是一种 regime"。

**对照现有代码**:

```python
# MOTION_REGIMES 是硬编码的 6 种 regime
# 没有"动态涌现新 regime"的机制
# 需要一个"regime 切换频率监控器"
```

**结论**: 概念好，但当前优先级低。当 Morphosphere 被应用于金融数据（对抗性环境）时才真正需要。

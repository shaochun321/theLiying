# 纠正: 运动势的真正来源

## 我之前错在哪里

```
我之前说的:
  运动势 = 一个预定义的 7 项复合公式
  Φ = Σ κ_a D² + λ_L D_L + ... (各种显式距离和权重)
  
  这是错的。

  我把运动势当成了一个人为写好的公式。
  实际上运动势是从物理结构中涌现出来的。
```

---

## 纠正 1: 没有显式坐标和距离

```
项目的铁律:
  系统内不能有显式坐标。
  系统内不能有显式距离。
  
  所有的"坐标"和"距离"都由结构本身蕴含:

  "距离" = 信号从元单元 A 到元单元 B 需要多少时间 (管道延迟)
         + 信号到达时衰减了多少 (管道衰减)

  没有一个变量叫 distance = 3.5。
  有的是: cable_delay = 3 ticks, attenuation = 0.7。
  
  距离不是被"计算"出来的——距离就是信号的到达时间和衰减量。
  它不是对物理距离的测量——它就是物理距离本身。
```

```
类比:
  你不知道两座城市之间有多少公里。
  但你知道从 A 城寄信到 B 城要 3 天, 信到了之后内容模糊了 30%。
  
  "3 天" 就是距离。"模糊 30%" 就是距离。
  不需要另外测量公里数。
  
  这就是项目说的: 时序和衰减就是时空测度。
  不是时空测度的代理——就是时空测度本身。
```

### 代码中已经有了这个

```
MetaSynapticBundle:
  cable_length → 物理长度
  propagation_velocity → 传播速度
  → delay = cable_length / propagation_velocity
  → 信号到达时间 = 延迟
  
  attenuation = exp(-cable_length × absorption_coeff)
  → 信号到达强度 = 衰减

  这两个量就是时空测度。
  代码已经有了!
  
  但代码没有意识到它们就是时空测度——
  代码把它们当作"管道参数",
  而不是"坐标系统的基础"。
```

## 纠正 2: 运动势从哪里来

### 因果链

运动势不是一个写好的公式。它是从下往上涌现的:

```
第 1 步: 物理结构
  元单元之间有管道 → 管道有延迟和衰减
  → 这些延迟和衰减构成了隐式的时空结构
  → 不需要显式坐标

第 2 步: 早期训练分离运动状态
  项目早期用真实物理数据训练
  → 系统学会区分: 运动 / 静止 / 旋转 / 等
  → 这些是无语义的纯物理运动状态
  → 不是"这是一只猫在走"——而是"这是运动模式 A"

第 3 步: 每种运动状态对应一种信息分布
  当系统处于"运动"状态时:
    元单元 1 激活 0.8, 元单元 5 激活 0.3, ...
  当系统处于"静止"状态时:
    元单元 1 激活 0.1, 元单元 5 激活 0.7, ...
  
  = 每种运动状态 → 一组元单元激活模式
  = 运动状态本身没有"量" (不知道走了多远/转了多快)
  = 量由元单元组合中不断变化的信息提供

第 4 步: 信息变化 → 时空轨迹
  信息在元单元组合中变更
  → 这些变更有时序 (先变这个, 后变那个)
  → 时序 + 元单元之间的隐式距离 (延迟+衰减)
  → = 信息的时空轨迹

第 5 步: 时空轨迹 → 时空测度变化
  轨迹穿过不同区域
  → 在不同时间窗口中的占比不同
  → 占比 = 时空测度

第 6 步: 时空测度变化 → 运动势
  测度变化的梯度
  → 信息"倾向于往哪里走"
  → 这就是运动势

第 7 步: 运动势 → 窗口占比和稳定性
  信息在某个时空窗口的占比稳定
  → 该区域是"稳定区域" (P 候选)
  长/跨窗口占比持续稳定
  → 该区域有高概率成为 P 锚点
  占比竞争
  → 不同区域的信息分布在竞争 → P 和 R 的竞争
```

### 完整因果链

```
物理结构 (延迟+衰减)
  ↓
隐式时空
  ↓
早期训练 → 运动状态分离 (无语义)
  ↓
运动状态 → 元单元激活模式
  ↓
信息在元单元组合中变更
  ↓
信息的时空轨迹 (变更的时序 × 隐式距离)
  ↓
时空测度变化 (轨迹在不同窗口的占比)
  ↓
运动势 (测度变化的梯度 = 信息倾向于往哪走)
  ↓
窗口占比 (信息在时空窗口中待了多久)
  ↓
稳定区域 (跨窗口持续占比)
  ↓
概率/竞争性 (P 和 R 的竞争)
```

## 纠正 3: 运动状态是什么

```
不是语义标签:
  ❌ "这是猫走路"
  ❌ "这是水流动"
  ❌ "这是温度上升"

是纯物理运动状态:
  ✅ 运动 (物质在位移)
  ✅ 静止 (物质不位移)
  ✅ 旋转 (物质在旋转)
  ✅ 等等...

运动状态没有量:
  "运动"不告诉你速度是多少。
  "旋转"不告诉你角速度是多少。
  
量由什么提供:
  元单元组合中不断变化的信息。
  激活模式的具体数值 = 量。
  运动状态的类型 = 质。
  
  质 + 量 = 完整描述
```

## 纠正 4: "测度值"从哪来

```
不是从公式计算出来的。
是从物理结构在时空窗口切片中的关系提供的。

"物理结构在长/跨窗口时空的切片关系":

  时空窗口 1 的切片:
    元单元 [1,3,7] 活跃, 其余沉默
    管道 [1→3] 权重高, [3→7] 权重中
    
  时空窗口 2 的切片:
    元单元 [3,7,12] 活跃
    管道 [3→7] 权重高, [7→12] 权重中
    
  跨窗口关系:
    信息从 [1,3,7] 移动到了 [3,7,12]
    → 元单元 1 退出, 元单元 12 加入
    → 信号沿 1→3→7→12 的方向流动
    → 这个方向 + 这个速度 = 运动势

  不需要一个公式来计算运动势。
  运动势就是信息分布在连续窗口切片中的移动方向和趋势。
```

## 我的修正理解

```
之前: 运动势 = 一个预定义的公式 Φ = Σ 各种项
现在: 运动势 = 信息在物理结构中的时空轨迹所涌现的运动趋势

之前: 时空测度 = 一个度规公式 s² = ...
现在: 时空测度 = 管道延迟和衰减本身, 不是它们的代理

之前: 运动状态 = 被分类的语义标签
现在: 运动状态 = 无语义的物理运动质 (运动/静止/旋转)
      量由元单元中的信息变化提供

之前: 公式先定义, 然后系统按公式运行
现在: 物理结构先存在, 然后运动势从结构中涌现
      公式只是对涌现结果的后验描述
```

> [!IMPORTANT]
> **核心纠正:**
> 
> 运动势不是"被设计的"——它是"被发现的"。
> 
> 早期训练分离出运动状态 → 每种状态对应信息分布 →
> 分布变化的轨迹在物理结构(延迟+衰减)中展开 →
> 轨迹的方向和趋势就是运动势。
> 
> 公式只是对这个涌现过程的数学描述,
> 不是驱动它的原因。
> 物理结构和真实运动才是原因。

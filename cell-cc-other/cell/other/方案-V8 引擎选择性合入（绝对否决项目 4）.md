### 一、决策总表

| 项目 | 内容 | 决策 | 理由 |
|------|------|------|------|
| 1 | Memristor 负电阻 fix | ✅ **合入** | 纯防御性硬件保护，无行为副作用 |

| **4** | **热编码非脉冲 + CRI 参数变更** | ❌ **绝对否决** | 与本地 Phase 1-3 已验证的配置冲突，风险不可接受 |

### 二、为什么绝对否决项目 4

项目 4 包含三项具体改动：

| 子项 | 本地状态 | 远端值 | 风险评估 |
|------|---------|--------|----------|
| `bc_current` | 0.02（经 Phase 1-3 验证）| 0.005 | 改变 Encoding 偏置会影响 T2.2/EXP-016 的全部数据 |
| `_thermal_encoding_config` | 本地已实现（Phase 1）| 远端也有 | 并非新功能，替换为远端版本可能丢失本地修复 |
| CRI 参数（q=0.02, R=1.0）| 本地已校准（Phase 2）| 远端相同值 | 值相同，但合入风险在于整体替换 `hebbian.py` 结构 |

**否决的核心原因**：
- 本地 Phase 1-3 的全部验证（EXP-016, EXP-017, EXP-019）都基于当前的 `hebbian.py` 配置。
- 远端的 `hebbian.py` 结构与本地存在差异，项目 4 不是简单的“参数替换”，而是“结构替换”。整体替换会引入无法预见的交互效应。
- **风险/收益比不成立**：没有证据表明远端参数优于本地已验证的参数。

### 三、为什么采纳项目 1-3

| 项目 | 类型 | 与本地的关系 |
|------|------|-------------|
| 1 | Memristor 负电阻 fix | **独立防御**，添加到现有 `semiconductor.py`，不改变任何行为 |
| 2 | LangevinNoise | **新文件**，不影响任何现有代码 |
| 3 | Langevin 注入 | **插入式**，在 `variant_adapter.step()` 中添加一行，替代机械正弦波 |

这三个项目都是**添加型**的，不修改本地已验证的核心逻辑。它们的风险极低，收益确定。

### 四、执行框架

#### 步骤 1：Memristor 负电阻 fix

**文件**：`semiconductor.py`

**操作**：
1. 在 `resistance` 属性中添加 `w_safe` 钳位：`w_safe = max(0.0, min(1.0, self.w))`
2. 在 `conductance` 属性中添加双重保护：`g = 1.0 / max(resistance_val, self.r_min)` 和 `min(g, 1.0 / self.r_min)`

**预期**：无行为变化，仅防止数值溢出

#### 步骤 2：LangevinNoise 组件

**文件**：`langevin_noise.py`（新文件，复制到 `components/`）

**操作**：
1. 创建新文件，实现 Ornstein-Uhlenbeck 过程的类
2. 关键参数：`sigma=0.07`（目标噪声强度），`theta=0.1`（回归系数），`dt=1.0`
3. 输出：`eta_x, eta_y, eta_z`（有色噪声）

**预期**：新文件，无现有代码被修改

#### 步骤 3：Langevin 注入 variant_adapter

**文件**：`variant_adapter.py`

**操作**：
1. 在文件顶部添加 `from components.langevin_noise import LangevinNoise`
2. 在 `__init__` 中添加 `self._langevin = LangevinNoise()`
3. 在 `step()` 中，找到 `mechanical_inputs['oto_x']` 赋值处
4. **替换机械正弦波**为 Langevin 噪声注入

**注意**：
- 不修改 `self.agc` 链路（Phase 4）
- 不修改 `process_hunger` 调用（Phase 1+4）
- 仅替换 oto_x 的输入源

#### 步骤 4：验证

**回归测试**：
```
python -m pytest nexus_v1/tests/test_regression.py -v
```

**AGC 验证**（确认 Phase 4 行为未退化）：
```
python -m pytest nexus_v1/tests/diag_agc_validation.py -v
```

**长程演化的行为观察**：
运行 50k 步（EXP-021 一半时程），观察 Δx 是否仍为正值。

### 七、三问确认

**Q1：为什么绝对否决项目 4？**

因为本地 Phase 1-3 的全部验证（EXP-016/017/019）都建立在当前 `hebbian.py` 配置上。远端的参数差异没有提供更强的实验证据，替换风险不可接受。

**Q2：项目 1-3 会破坏 Phase 4 AGC 吗？**

不会。项目 1 是硬件保护层，项目 2 是新文件，项目 3 仅替换 oto_x 的输入源（机械正弦波 → Langevin 噪声），不触及 AGC 链路和 hunger reflex。

**Q3：执行后如何确认系统仍在正常工作？**

三步验证：回归测试（12/12 PASS）→ AGC 验证（5/5 PASS）→ 50k 长程运行（Δx > 0）。全部通过即可确认。
